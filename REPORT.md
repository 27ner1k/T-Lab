# Отчет по заданию "Аналитика над графом"

## 1. Выбранная метрика: PageRank

PageRank - это алгоритм ссылочного ранжирования, разработанный основателями Google Ларри Пейджем и Сергеем Брином в конце 1990-х годов. Он используется для оценки "важности" или "авторитетности" веб-страниц на основе количества и качества гиперссылок, ведущих на них. 


### Формула
    PR(v) - важность вершины v

    PR(v) = (1 - d) / N + d * (sum(PR(u) / out_degree(u)) + dangling_sum / N )

где:
- сумма берется по всем вершинам `u`, из которых есть ребро в `v`;
- `d = 0.85` - коэффициент затухания (стандартное значение);
- `N` - число вершин;
- `dangling_sum` - сумма рангов вершин без исходящих ребер; она перераспределяется равномерно между всеми вершинами, иначе ранг "утекал" бы из системы и сумма рангов не сохранялась.

### Почему PageRank

1. **Устойчивость к гипер-узлам.** Вклад вершины делится на ее исходящую степень: вершина с 50 000 исходящих ребер отдает каждому соседу 1/50000 своего ранга и не искажает картину.
2. **Естественно ложится на считывание по частям.** Одна итерация - линейный проход по ребрам, порядок ребер не важен. Это позволяет читать граф с диска блоками, не загружая его в память. Из-за этого данную метрику легче масштабировать, чем другие.
3. **Есть эталонная реализация** (`networkx.pagerank`) для строгой проверки корректности. Также этот алгоритм достаточно прост в реализации, к тому же, можно сверить код с уже написанным.

## 2. Варианты представления графа

### Вариант 1. Бинарный edge list на диске + стриминг блоками - ВЫБРАН

Ребра хранятся на диске в бинарном файле как последовательность пар `(int32 from, int32 to)` в перенумерованных id, где количество id = количество вершин. На каждой итерации файл читается блоками по 65 536 ребер (512 КБ), вклады ребер накладываются на вектор рангов в RAM. В памяти постоянно живут только массивы размера O(V). (V - количество вершин, E - количество ребер)

**Плюсы:** память O(V) и не зависит от E; последовательное чтение - оптимальный выбор для SSD, блок ребер обрабатывается параллельно.
**Минусы:** каждая итерация - полный проход по файлу; запись вкладов в `new_rank[to]` требует синхронизации потоков с помощью atomic. Возникает гонка данных: несколько потоков могут одновременно обновлять одну и ту же ячейку `new_rank[to]`, из-за чего вычисления будут неверны. Это устраняется с помощью atomic. Он немного теряет в скорости, но зато сохраняет память, которая так важна для задания.

### Вариант 2. CSR в памяти - отказ

Классическое представление: массив, где индекс i соответствует вершине i, размера V+1 и массив соседей размера E. Дает быстрый доступ к соседям и хорошее расположение в памяти, которое дает быструю производительность.

**Отклонен:** требует O(V + E) памяти. По условию граф (1 ГБ) не помещается в RAM (128 МБ) - CSR не решает поставленную задачу в принципе. 

### Вариант 3. Партиционирование ребер по вершине-назначению - отказ

Ребра предварительно сортируются (внешней сортировкой) по `to` и разбиваются на куски. Каждый поток обрабатывает свой диапазон вершин-назначений - синхронизация не нужна вовсе.

**Отклонен:** требует внешней сортировки 69M ребер и заметно усложняет код. Выигрыш от отказа от atomic не окупает сложность на данных такого размера.

## 3. Схема памяти и расчет

### Что где хранится

| Структура                        | Размер               | Где  |
|----------------------------------|----------------------|------|
| Ребра (пары int32)               | 8 байт * E           | Диск |
| `vertex_rank`, `new_vertex_rank` | 8 байт * V каждый    | RAM  |
| `degree_vertex`, `to_back_id`    | 4 байта * V каждый   | RAM  |
| `to_new_id`                      | 4 байта * (max_id+1) | RAM  |
| Буфер чтения                     | 512 КБ               | RAM  |

### Расчет для LiveJournal (V ≈ 4.85M, E ≈ 69M, max_id ≈ 4.85M)

- `vertex_rank` + `new_vertex_rank`: 2 * 4.85M * 8 ≈ **78 МБ**
- `degree_vertex` + `to_back_id`   : 2 * 4.85M * 4 ≈ **39 МБ**
- `to_new_id`                      : 4.85M * 4     ≈ **19 МБ** (освобождается после конвертации)
- буфер:                                             **0.5 МБ**

**Пик ≈ 130–140 МБ** и не зависит от числа ребер. Сами ребра (69M * 8 ≈ 550 МБ бинарно) лежат на диске. При дальнейшем росте графа по ребрам потребление RAM не меняется вовсе; по вершинам - растет линейно с малым коэффициентом (24 байта на вершину в фазе итераций).

### Проверка ограничения памяти на практике

Программа запускалась внутри Docker-контейнера с жестким лимитом RAM (`--memory --memory-swap`, swap отключен) на LiveJournal:

| Лимит контейнера | Результат | Время |
|------------------|-----------|-------|
| 256 МБ           | Успех (exit 0, 4 847 572 строк) | ~2 минуты |
| 224 МБ           | OOM на этапе финальной сортировки |  |
| 192 МБ и ниже    | OOM |  |

То есть программа успешно обрабатывает граф, CSV которого весит ~1 ГБ, в контейнере с 256 МБ RAM - CSV в ~4 раза больше доступной памяти. Пик приходится не на сами итерации PageRank (~130-140 МБ), а на финальную сортировку результата: временный `std::vector<std::pair<int32_t, double>>` размера V добавляет еще ~58 МБ.

### Ключевое решение: перенумерация через плотный вектор, а не unordered_map

Исходные id вершин не обязаны быть плотными, поэтому нужна перенумерация в [0..V). Наивный вариант - `std::unordered_map<int32_t, int32_t>` - на 4.85M вершин занял бы сотни МБ из-за накладных расходов хэш-таблицы.

Вместо этого: **два прохода по CSV**. Первый проход находит `max_id` (память O(1)), второй строит перенумерацию в плотном `std::vector<int32_t>` размера max_id+1. Цена - одно дополнительное чтение CSV, но зато - сотни МБ памяти и более быстрое получение вершин.

Ограничение подхода: если id разреженные (например, 1000 вершин с id до 2^31), плотный вектор станет невыгоден - тогда правильным выбором была бы хэш-таблица. Для датасетов SNAP id плотные, поэтому выбран вектор.

## 4. Параллельная обработка

Все тяжелые циклы распараллелены через OpenMP:

1. **Суммирование dangling-рангов** - `reduction(+:dangling_sum)`.
2. **Инициализация new_rank базовым значением** - независимые записи.
3. **Накопление вкладов ребер** - основной цикл; несколько потоков могут писать в один `new_rank[to]` (особенно для гипер-узлов), поэтому используется `#pragma omp atomic`.
4. **L1-норма разницы** для критерия сходимости - `reduction(+:diff)`.

### Выбор между двумя способами параллельного накопления

Альтернатива atomic - каждому потоку своя копия вектора рангов с финальной редукцией. Это убирает конкуренцию, но стоит O(V) памяти на поток: при 16 потоках и 4.85M вершин - много лишних МБ, что нарушает бюджет памяти. Выбран atomic: конкуренция реальна только на гипер-узлах, а узким местом все равно является чтение с диска, а не синхронизация CPU. Это осознанный выбор: память в обмен на скорость, в пользу памяти, как и нужно в задании.

## 5. Сложность и сходимость

- **Одна итерация:** O(V + E) - три прохода по вершинам и один по ребрам.
- **Всего:** O(k · (V + E)), где k - число итераций до сходимости.
- **Дисковый I/O:** k последовательных чтений бинарного файла (8E байт) + два чтения исходного CSV.
- **Худший случай = средний случай:** алгоритм не зависит от структуры графа, все проходы линейные.

### Сходимость

Итерация PageRank - сжимающее отображение с коэффициентом d = 0.85 (стандартный результат теории цепей Маркова: матрица переходов стохастическая, d < 1). Ошибка убывает геометрически: ||x_k − x*|| <= 0.85^k · ||x_0 − x*||. Для порога < 1e-10 достаточно k ≈ log(1e-10)/log(0.85) ≈ 140 итераций. Критерий остановки: норма разности соседних приближений; при недостижении порога - жесткий лимит 1000 итераций.

### Детерминированность

Случайность в алгоритме отсутствует: фиксированная инициализация 1/N, детерминированный порядок чтения ребер. Единственный источник недетерминизма - порядок сложения double при параллельном atomic-накоплении; он дает расхождения на уровне эпсилона (наблюдаемо ~1e-15 на ранг), что на порядки ниже порога сходимости и допуска сравнения. Повторные запуски дают совпадающие результаты в пределах 1e-10.

## 6. Проверка корректности

Эталон - `networkx.pagerank` с ужесточенным порогом (`tol=1e-12`, `max_iter=200`; по умолчанию у networkx tol=1e-6, что грубее моего 1e-10 - сравнение было бы нечестным).

Результаты на тестовых графах (максимальное поэлементное расхождение с эталоном):

| Тест | Граф                                  | max_diff |
|------|---------------------------------------|----------|
| 1    | базовый                               | 1.3e-08  |
| 2    | dangling-вершины                      | 0        |
| 3    | изолированные вершины                 | 0        |
| 4    | петли                                 | 0        |
| 5    | одно ребро                            | 0        |
| 6    | web-Google (876K вершин, 5.1M ребер)  | ~1e-08   |
| 7    | LiveJournal (4.85M вершин, 69M ребер) | ~1e-08   |

Расхождения порядка 1e-8 - суммарный эффект конечной точности double и различий порогов сходимости двух реализаций; это ожидаемый уровень для корректного алгоритма.

Тестовые графы 2–5 покрывают крайние случаи: dangling-вершины, изолированные вершины, петли (self-loop), минимальный граф.

## 7. Производительность

LiveJournal (69M ребер), Docker, 16 логических ядер, SSD: **~2.5 минуты** полный цикл - конвертация CSV, итерации до сходимости, запись результата. С лимитом контейнера 256 МБ время не растет (~2 минуты).

## 8. Ограничения подхода

1. **Один вычислительный узел** - по условию; для графов, где даже O(V)-массивы не влезают в RAM, потребовалось бы партиционирование вершин.
2. **id вершин - int32** - по формату входных данных из условия.
3. **Плотный вектор перенумерации** невыгоден при сильно разреженных id.
4. **Atomic на гипер-узлах** создает конкуренцию потоков - осознанный компромисс в пользу памяти.
5. **Два прохода по CSV** на этапе конвертации - цена отказа от хэш-таблицы.
6. **Финальная сортировка результата** удваивает пик памяти на V - при более жестких лимитах памяти можно перейти на in-place сортировку по индексам.
